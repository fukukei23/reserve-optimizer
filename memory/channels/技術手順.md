### ⚠️ コンテナの制約（2026-03-15確認）

- **コンテナ内からDocker操作はできない**
  - dockerコマンドが入っていない
  - Docker socketはあるが権限なし（nodeユーザー uid=1000）
  - sudoも入っていない
- **Dockerビルド・コンテナ再作成はVPS上で直接実行する**
- この手順はふくけいさんがVPSにSSH接続して実行する必要がある

### Docker構成方針（2026-03-15）

**現在の構成**:
- ベースイメージ: `ghcr.io/openclaw/openclaw:2026.3.12`
- Chromium: 手動apt install済み（`/usr/bin/chromium`）
- 設定: `browser.executablePath: /usr/bin/chromium`

**公式Dockerfileに切り替えない理由**:
1. 現在のChromium構成が動作確認済み
2. Playwright経由になるとパスが変わる可能性
3. `openclaw.json`の再設定が必要
4. 追加パッケージは現在のDockerfileに追記で対応可能

**パッケージ追加時**: 現在のDockerfileの`apt-get install`行に追記する

### ブラウザ自動化の解決（2026-03-15）

**原因**: コンテナ内ではChromiumのサンドボックスが使えない
**解決策**: `openclaw.json`に`"noSandbox": true`を追加

```json
"browser": {
  "executablePath": "/usr/bin/chromium",
  "headless": true,
  "noSandbox": true
}
```

**結果**: browser toolが正常に起動するように（running: true, cdpReady: true）

### ブラウザ操作の使い分け（2026-03-15）

**コンテナ内Chromiumでできる**:
- ログイン不要のサイトの閲覧・操作
- 新規ログイン（ユーザー名/パスワードのみ）
- スクレイピング・情報収集
- フォーム入力・送信
- スクリーンショット

**Surface Go + Browser Relayが必要**:
- 2FA（二要素認証）が必要なサイト
- Cookieベースの既存ログイン状態を維持したい場合
- 特殊認証フロー（CAPTCHA、デバイス認証など）

### 自動マネタイズの方針（2026-03-15）

**推奨**: API優先で設計し、必要になったらブラウザに切り替え

理由：
1. APIは安定（UI変更に影響されない）
2. 高速・安価
3. 再利用性が高い
4. 二度手間にならない（APIロジックは流用可能）

**流れ**:
1. まずAPIで実装
2. API制限にぶつかったらブラウザに切り替え
3. 2FAが必要ならSurface Go起動

**Surface Go**: 「いつでも使える」状態にしておく（常時起動不要）
