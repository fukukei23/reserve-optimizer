## 推奨シーケンス（依存を解決した実行順）

| Phase | タスク | 意図 |
|---|---|---|
| **α 土台・導入準備**（安く必須） | A4(セキュリティ) → C4(規約) → C2(料金) → D1(状態機械) → A1(テスト) → C5(監視) | 実導入前に**必須の最小セット**・D1は機能増やす前の土台 |
| **β 実導入** | C1(営業→実データ) → A2(数値化) | 全ての価値の源泉・A2は実データで真値化 |
| **γ 差別化機能** | B2(カレンダー同期) → A5(図) → D2(Worker安定化) | ピボット先直結の差別化＋採用アピール |
| **δ SaaS化** | D3/D4(DBIF/Supabase設計) → B4(マルチテナント) | スケール前提・土台から |

> **Phase α（A4+C4+C2+D1+A1+C5）** は「工数S-M×6で導入可能状態を作る」最小必須セット。

## 依存関係メモ（順位付け時の参考）
- **A2（Impact数値化）は C1（実導入）に依存** — 実データが無ければベンチマーク値になる
- **B4（マルチテナント）は D3/D4（Supabase/DBIF）に依存** — Sheets 構成のままでは困難
- **C2（料金）・C4（規約）は C1（導入）の前置き** — 販売前に確定が必要
- **A4（セキュリティ）は C1 前に最低限必要**（CORS/キー放置での公開は危険）
- **D1（状態機械）は B 系追加すべての前置き** — これをやらずに機能増やすと複雑度爆発
